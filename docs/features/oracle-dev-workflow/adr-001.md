# ADR-001: Zero-dependency Node state & gate engine

**Status:** Accepted
**Date:** 2026-07-08
**Deciders:** Philippe (contract signatory), Claude (implementer)

## Context

The plugin needs deterministic, mutation-testable logic for state parsing and gate evaluation. The signed contract constrains the design: gates are read-only and only advancing writes (C12); the meta-plugin routes to the seven underlying plugins and inlines nothing they provide (C13); contracts are per-feature with a root fallback (C19); the verify gate must read StrykerJS output with a raw, no-rounding comparison (C6/C7). The test-oracle plugin's stack is Vitest + fast-check + StrykerJS, which only mutates JS/TS — so the engine's language choice decides whether the plugin can pass its own verify gate.

## Decision

Build the engine as zero-runtime-dependency Node ESM modules: pure functions in `scripts/lib/`, thin CLI entrypoints (`board.mjs`, `gate.mjs`) that do I/O at the edge. State lives in committed markdown frontmatter (`docs/features/<slug>/workflow.md`) parsed by a hand-rolled minimal frontmatter parser. Dev-dependencies (Vitest, fast-check, StrykerJS) are test-only.

## Options considered

### Option A: Zero-dep Node ESM + hand-rolled frontmatter parser (chosen)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low–Med (parser is ~60 lines for the flat subset we need) |
| Cost | None at runtime; Node ≥ 18 assumed (ships with Claude Code environments) |
| Scalability | Fine — dozens of features per repo, trivial I/O |
| Mutation-testability | Excellent — pure functions, Stryker-native |

**Pros:** installs nowhere, breaks nowhere; the parser itself is a prime oracle target (round-trip property); Stryker applies directly, so the plugin can pass its own verify gate.
**Cons:** flat-subset YAML only (string values, no nesting) — a deliberate format restriction we must document.

### Option B: Bash scripts

**Pros:** zero toolchain. **Cons:** not mutation-testable by the test-oracle stack — the plugin could never pass its own verify gate. Rejected on self-hosting grounds alone.

### Option C: Node + `yaml` dependency

**Pros:** full YAML. **Cons:** runtime dependency in every target repo for a format subset we don't need; the dependency's parser is untestable-by-us surface. Rejected as scope beyond the contract's needs.

### State location: committed markdown frontmatter vs `.claude/` JSON

Frontmatter in `docs/features/` wins: team-visible in PRs, human-editable, and C18's adopt-and-backfill behavior assumes the record is repo content. `.claude/*.local.md` remains for per-repo settings only (threshold override), per the plugin-settings convention.

## Trade-off analysis

The binding constraint is self-hosting: the verify gate demands a Stryker report (C7) at ≥ 80 raw (C6), so the engine must be Stryker's substrate — Node. Given Node, zero-dep vs `yaml` trades a documented format restriction for a smaller, fully-owned oracle surface; the contract's state needs (flat string fields) make the restriction cheap.

## Consequences

- Easier: mutation testing the whole engine; C8 traceability (tests tag clauses directly over pure functions); running gates headless later (CI, v2).
- Harder: state frontmatter must stay flat — nested metadata would force a parser change (documented in the state reference).
- Revisit: if v2 hooks need richer state, reconsider a structured sidecar file rather than nesting YAML.
